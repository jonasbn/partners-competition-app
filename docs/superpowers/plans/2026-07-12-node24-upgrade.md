# Node.js 24 Upgrade Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make Node.js 24 (current Active LTS, "Krypton") the primary supported version for local development, CI, and the Digital Ocean production deployment — replacing Node 22 as primary while continuing to test against it in the CI matrix.

**Architecture:** This is a version-pin bump, not a code change — a same-day local smoke test (full suite + production build under Node 24.18.0) already confirmed the application and its dependencies (Vite 8 / rolldown, Vitest 4, jsdom) run correctly with no source changes required. The work is: (1) bump `.nvmrc`/`package.json` engines for local dev, (2) bump the CI workflows' single-pinned-version jobs from `22.x` to `24.x` (the matrix jobs already cover `[22.x, 24.x]` — see Global Constraints), (3) validate via a PR before merging, (4) merge to `main` and confirm the Digital Ocean buildpack picks up the new version and deploys successfully.

**Tech Stack:** Node.js, npm, GitHub Actions, Digital Ocean App Platform (Heroku-derived `heroku/nodejs` buildpack), Vite, Vitest.

## Global Constraints

- Node 24.x is current Active LTS (EOL 2028-04-30); Node 22.x is now Maintenance LTS (EOL 2027-04-30) — both are supported, so CI should keep testing both, not drop 22.x.
- **Correction to stated premise:** CLAUDE.md and README.md currently claim "CI tests against Node 20.x and 22.x," but the actual workflow files (`.github/workflows/ci.yml:15`, `.github/workflows/test.yml:51`) already use a matrix of `[22.x, 24.x]` — there is no `20.x` anywhere in `.github/workflows/`. The docs are stale, not the workflows. This plan corrects the docs and shifts the *single-pinned-version* jobs (which aren't part of the matrix) from `22.x` to `24.x`; it does not need to "remove 20.x" from any workflow file because it isn't there.
- Node 24 bundles **npm 11.x** (confirmed locally: `node v24.18.0` → `npm v11.16.0`), not npm 10.x. `package.json`'s `engines.npm` must move from `10.x` to `11.x` alongside the node bump, or the Digital Ocean buildpack will attempt an unnecessary npm downgrade-swap on every build (the same class of failure fixed for the `>=` wide-range issue on 2026-07-12 — see commit `9e9f716`).
- Confirmed locally under Node 24.18.0 / npm 11.16.0: `npm install` succeeds (383 packages, 0 vulnerabilities), `npm run test:run` passes 231/231, `npm run build` succeeds. No application source code changes are needed.
- Digital Ocean's `.do/partners-competition-app.yaml` pins `buildpack-stack=ubuntu-22`; it reads the Node version from `package.json` `engines.node` (already confirmed working for the `22.x` pin in commit `9e9f716` — deployment `06c97246`, ACTIVE). This plan relies on that same mechanism for `24.x` — no changes to `.do/partners-competition-app.yaml` are needed.
- Do not touch `App.jsx`, any component under `src/components/`, or `src/utils/tournamentUtils.js`/`dataUtils.js` — this is purely a tooling/version bump, no application logic changes.
- Work happens on a feature branch (`chore/upgrade-node-24`) with a PR, so CI validates the bump (matches the user's requested "then we can see if the application works" checkpoint) before it reaches `main` and triggers the real Digital Ocean deploy-on-push.

### Task 2: Bump CI workflows' pinned Node version from 22.x to 24.x

**Files:**
- Modify: `.github/workflows/ci.yml`
- Modify: `.github/workflows/test.yml`
- Modify: `.github/workflows/quality.yml`

**Interfaces:**
- Consumes: nothing from Task 1 directly (workflow YAML `node-version` values are independent of `.nvmrc`/`package.json engines`).
- Produces: CI jobs running on Node 24.x as the canonical/single-pinned version, with the existing `[22.x, 24.x]` matrix jobs unchanged (already correct). Task 3 depends on these files being correct before the PR is opened.

- [ ] **Step 1: Update `.github/workflows/ci.yml`'s single-pinned-version jobs**

This file has one matrix job (`test`, already `[22.x, 24.x]`) and four single-version jobs (`accessibility-and-performance`, `security`, `code-quality`, `deployment-preview`) each pinned `node-version: '22.x'`. The literal string `node-version: '22.x'` (quoted, single-version form) appears exactly 4 times in this file — all 4 are the single-version jobs; the matrix job uses the distinct string `node-version: [22.x, 24.x]` (array form, unquoted values), so a blanket replace of the first string cannot touch the matrix line. Confirm this before editing:

```bash
grep -c "node-version: '22.x'" .github/workflows/ci.yml   # expect: 4
grep -c "node-version: \[22.x, 24.x\]" .github/workflows/ci.yml   # expect: 1
```

Replace all 4 occurrences of the single-version string:

```bash
sed -i '' "s/node-version: '22.x'/node-version: '24.x'/g" .github/workflows/ci.yml
```

Verify: `grep -c "node-version: '22.x'" .github/workflows/ci.yml` now reports `0`, and `grep -c "node-version: \[22.x, 24.x\]" .github/workflows/ci.yml` still reports `1` (matrix array untouched).

Also update the artifact-upload condition — the build artifact should now be captured from the `24.x` matrix leg (the new canonical version) instead of `22.x`. Change:

```yaml
    - name: Upload build artifacts
      uses: actions/upload-artifact@043fb46d1a93c77aae656e7c1c64a875d1fc6a0a # v7.0.1
      if: matrix.node-version == '22.x'
```

to:

```yaml
    - name: Upload build artifacts
      uses: actions/upload-artifact@043fb46d1a93c77aae656e7c1c64a875d1fc6a0a # v7.0.1
      if: matrix.node-version == '24.x'
```

Leave `strategy: matrix: node-version: [22.x, 24.x]` (around line 15) exactly as-is — it already tests both currently-supported LTS versions correctly.
